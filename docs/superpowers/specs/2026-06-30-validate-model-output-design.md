# 설계: `validate-model-output` — 수치모델 후처리·검증 자동화 스킬

- 날짜: 2026-06-30
- 팀/작성자: A팀 / Youjung Oh (오유정)
- 대회: 예보사업부 AX 해커톤 (공통과제 = 반복 수작업 1개 자동화)
- 채점 나침반: 과정 70(①활용깊이25 ②**재현성·전파성25** ③전원참여10 ④문제정의10) + 결과 30(⑤완성도15 ⑥임팩트12 ⑦자유3). 동점 시 **②→①→③** 우선.

## 1. 문제 정의

수치모델·AI모델의 출력을 **후처리하고 검증**하는 일을 하루에도 여러 번 반복한다. 출력은 NetCDF/GRIB/텍스트 등 포맷이 다양하고, **규약이 제각각**이라(좌표 1D/2D, 단위 K/°C, 변수명 ECMWF식/GRIB식) 지금은 CDO/NCO·Python·ncview·wgrib2를 **손으로 오가며 눈으로** 확인한다. 느리고, 세션마다 절차가 흔들리며, 통과/실패 근거가 안 남는다.

## 2. "맨손 Claude"와의 차별 (이 설계의 존재 이유)

맨손 Claude의 약점은 ① 재현 불가(세션마다 검증 코드가 달라짐) ② 전파 불가(프롬프트가 머릿속에만) ③ 검증이 "눈으로". 대회 최고 배점이 **②재현성·전파성**이므로, 산출물은 **재사용 자산**이어야 한다.

**해법 = 결정적 코어 + 에이전트 껍질의 분업:**
- 흔들리면 안 되는 검증 규칙 → **결정적 Python 코드 + 선언적 `rules.yaml`**에 박제 (완전 재현)
- "어떤 포맷·어떤 모드·어느 파일이 기준·결과 해석·수정 제안" 같은 판단 → **에이전트**가 유도 대화로 처리
- 흔들리면 안 되는 곳은 코드로, 판단이 필요한 곳은 에이전트로. → ①활용깊이 + ②재현성 동시 충족.

## 3. 일반화 5원칙 (하드코딩 금지 — 특정 파일에 종속 X)

1. **입력 포맷 자동 감지**: 확장자/파일 시그니처로 NetCDF3 / NetCDF4·HDF5 / CSV / 텍스트 판별 → 맞는 reader로 라우팅. 새 포맷은 reader 추가(플러그인식).
2. **격자 규약 자동 적응**: 좌표가 1D(`lat/lon`)인지 2D(`y/x`+2D lat/lon)인지, 차원 이름이 무엇이든 자동 인식. 코드 분기 없이 같은 경로.
3. **규칙은 선언적 `rules.yaml`(유일한 확장점)**: 변수별 물리범위·결측임계·필수속성. 매칭은 변수명만이 아니라 `units`/`standard_name`/패턴으로 유연하게. **동료는 코드 안 건드리고 yaml만 추가**해 자기 자료에 적용.
4. **단위 인식 검증**: 같은 물리량이 K/°C로 와도 `units`를 읽어 올바른 한계 적용(또는 표준단위 정규화 후 검사). 규칙에 없는 변수는 크래시 대신 **WARN + 통계적 이상치만**.
5. **두 실데이터 = 일반성 증명 데모**: ERA5(1D좌표·K·CF명)와 GFS(2D좌표·°C·GRIB명)를 **같은 명령**으로 처리·검증. "포맷·규약을 안 가린다"가 라이브에서 보임.

## 4. 검증의 두 층위 (둘 다 구현 = 범위 B)

**층위 1 — 자료 품질 검증 (QC, intrinsic, 기준자료 불필요)**
파일 하나를 물리·형식 상식에 비춰 점검:

| 체크 | 내용 |
|---|---|
| 포맷/열림 | 정상 오픈·미잘림 |
| 격자/스키마 | dims/coords 존재, lat/lon 단조·해상도, (2D좌표면 형상 일치) |
| 값범위 | 변수별 물리 범위(units 인식) |
| 결측 | NaN/_FillValue 비율 ≤ 임계 (정당한 mask와 구분 시도) |
| 이상치 | 물리한계/N-시그마 초과 |
| 메타/시간축 | units·필수 attrs, 타임스탬프 단조·중복/누락 |

**층위 2 — 예보 검증 (forecast verification, 기준자료 대비)**
예보 파일을 기준자료(재분석/관측)와 비교해 오차 산출:
- **후처리 브리지(검증 전 필수)**: 단위 통일(°C↔K 등), 변수 매칭(t2m↔TMP, u10↔UGRD …), 격자 정렬(이미 동일 격자면 직접 비교; 다르면 정렬 필요 — 오늘 데이터는 721×1440 동일).
- **메트릭(핵심만)**: bias(평균오차), RMSE, MAE, (가능하면) 상관. 변수 기본값 = 기온·바람.
- **산출**: 메트릭 표 + 차이지도(diff field) 저장(옵션).
- 기준자료는 ERA5로 고정(데모). 재사용 시 yaml/인자로 변경 가능.

## 5. 후처리 (가벼움 — 후처리→검증 루프)

- **영역 자르기(subset)**: lat/lon 바운딩박스. 1D·2D 좌표 모두 지원.
- **파생변수**: u,v → 풍속 `sqrt(u²+v²)`.
- 후처리 산출물에 **QC 자동 재실행** → "처리해도 데이터 건강함" 증명.

## 6. 산출물 구조 (스킬 폴더 = 전파 단위)

```
skills/validate-model-output/
  SKILL.md                 # 자연어 진입점 + 유도 대화 흐름(핵심) + 사용법
  scripts/
    validate.py            # CLI: 포맷감지 → QC 실행 → 리포트
    verify.py              # CLI: 예보 vs 기준 → 단위·격자 정렬 → 메트릭 리포트
    postprocess.py         # CLI: subset / 파생변수 → 결과에 QC 재실행
    io_detect.py           # 포맷 감지 + reader 라우팅(원칙 1·2)
    checks/                # QC 체크 모듈(원칙 3·4의 결정적 코어)
    report.py              # JSON + 마크다운 리포트 생성
  rules.yaml               # 변수·단위별 범위·임계(유일한 확장점)
  data/                    # 소형 fixture: 정상본 + 고의 결함본(ERA5/GFS clip)
  tests/                   # pytest: 정상=PASS / 결함=FAIL 자동 보증
  references/usage.md      # 따라하기 가이드(재현성)
```

거대한 원본(598MB ERA5 / 1.14GB GFS)은 **git에 올리지 않음**. 작게 잘라낸 fixture만 포함.

## 7. SKILL.md 유도 대화 흐름 (에이전트 껍질의 가치)

```
사용자: "이 모델 출력 검증해줘"
 ① 의도 분류: QC(품질) vs 예보검증(기준 대비 오차)? — 모호하면 한 번 되물음
 ② 예보검증이면: 어느 파일이 예보/기준? (파일명 fcst·rean·era5·gfs로 추정·확인)
 ③ 변수: 기본(기온·바람) or 지정? / 영역·시각 전체 or subset?
 ④ 단위·격자 차이 자동 처리 안내 → 결정적 스크립트 실행 → 리포트 해석·수정 제안
```

비전문가가 "기준자료"를 몰라도 스킬이 되물어 올바른 검증으로 유도 → 전파성↑.

## 8. 리포트 형식

- `report.json`(기계용) + `report.md`(사람·심사위원용 표): 체크/메트릭별 **PASS/FAIL/WARN + 근거**(예: "t2m 12개 값 > 340K, index …", "lat 비단조").
- 검증기는 **절대 크래시하지 않고** 문제를 FAIL로 기록.

## 9. 고의 결함 데모 (60초 미니시연, 1인 라이브)

결함본은 일부러: 값범위 초과(기온 500K), 결측 구멍, 격자 비단조/중복, 파일 잘림/형상 어긋남.
1. `validate <clean>` → **전부 PASS**
2. `validate <broken>` → **FAIL: 값범위·결측·격자 — 각 근거와 함께 적발**
3. `verify <gfs> --ref <era5>` → 단위·격자 정렬 후 **bias/RMSE 표 + 차이지도**
4. `postprocess`(subset+풍속) → 결과에 QC 재실행 → PASS

"정상 통과 / 비정상 정확 적발 / 예보검증 / 처리 후 건강"을 한 흐름에. 맨손 Claude와 명확히 구분.

## 10. 테스트·에러 처리

- `tests/` pytest: 정상본=PASS, 결함본=각 체크 FAIL을 **자동 보증** → 재현성 증거(②).
- 에러: 파일없음/미지원포맷/손상 → 명확한 메시지, 크래시 없이 FAIL 리포트.

## 11. 전원 참여 반영 (③)

스킬은 **팀 공유 자산** — 동료들이 각자 다른 포맷·자료·모드로 본인 60초 시연 가능(한 명 QC, 한 명 예보검증, 한 명 결함적발). 본인(오유정) 개인 파트 = 검증 코어 설계 + 담당 트랙 + 결함적발/예보검증 데모.

## 12. 범위 밖 (YAGNI — 오늘 안 함)

- 해상도 보간(regrid, CDO/xesmf 의존) — 격자 다른 경우의 일반 재격자화는 후속.
- GRIB 네이티브 reader(wgrib2 의존, 보안 부담) — 후속 plug-in.
- 웹 UI/대시보드 — CLI + 마크다운 리포트로 충분.

## 13. 성공 기준

- 두 실데이터(ERA5/GFS)가 **같은 명령**으로 QC 통과, 결함본은 각 결함 적발.
- GFS를 ERA5로 예보검증해 bias/RMSE 산출.
- 동료가 `rules.yaml`만 수정해 새 변수/자료에 적용 가능(코드 무수정).
- pytest 그린, 리포트 재현 가능.
